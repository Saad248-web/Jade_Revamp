Project Architecture & AxisRooms Integration

Jade Hospitainment — ReVamp · Codebase analysis · Generated from actual src/ models, APIs, and dashboard modules · June 2026

Direct booking: live architecture AxisRooms / STAAH: stub only MongoDB sole database

1. Executive summary

Jade ReVamp is a Next.js 14 marketing site plus a custom villa PMS (Jade Host dashboard) with a direct booking engine, Razorpay payments, and MongoDB as the single operational database.

Important: The root README.md still mentions PostgreSQL. The running app uses MongoDB only — see src/lib/db.ts and NEEDS_FROM_USER.md. Legacy PG scripts are not used.

AxisRooms (replacing legacy STAAH ChannelConnect naming in code) is architecturally planned but not implemented. Outbound push, inbound webhooks, and API clients are stubs. Direct website bookings work end-to-end without a channel manager.

Public site

  • Villa marketing pages
  • /book booking wizard
  • Leads, careers, SEO

Jade Host PMS

  • Calendar & folio
  • Housekeeping & blocks
  • Staff RBAC (5 roles)

Operations DB

  • MongoDB Atlas / local
  • Bookings + night locks
  • Villas + CMS pages

Channel manager

  • Mapping fields on Villa
  • Sync flags on Booking
  • Retry cron (stub push)

7. AxisRooms — what must come from the channel manager

Direct bookings work without AxisRooms. AxisRooms is required for OTA distribution and two-way sync.

Code reality: Integration code is named staah (STAAH_API_KEY, staahSynced). UI and Villa schema reference Axis Rooms as the cutover target. There is no src/lib/axisRooms/ client.

Must obtain from AxisRooms

ItemWhy (code gap)
Property / room-type / rate-plan IDsVilla.axisRooms fields empty after seed
Inbound reservation webhooks/api/webhooks/staah is stub — no booking creation
Outbound reservation push APIpushStaahReservation() — audit log only
Outbound cancellation APIpushStaahCancellation() — stub
OTA source identifiersBooking.source enum exists; nothing writes OTA sources
API auth specificationAXIS_ROOMS_API_KEY checked; not in .env.example

Does NOT need AxisRooms (built locally)

  • Direct website booking + 15 min holds
  • Razorpay payment collection
  • Direct-booking pricing (computeBookingPricing)
  • PMS calendar, blocks, housekeeping, folio
  • Staff RBAC dashboard
  • Partial CMS (ContentPage + Villa.content)

13. AxisRooms requirements checklist (request from vendor)

  1. API documentation — reservation create/update/cancel and availability (push and/or pull), including sandbox access.
  2. Authentication — API key format, webhook Authorization header, IP allowlist, rate limits.
  3. Property mapping sheet — per Jade villa: propertyId, roomTypeId, ratePlanId; confirm 1:1 villa-to-room-type with inventory = 1.
  4. Inbound webhook contract — JSON schema for new/modified/cancelled reservations; idempotency key; retry policy.
  5. Outbound reservation API — push direct confirmed bookings so OTAs see reduced availability.
  6. Cancellation / release API — notify when direct booking cancelled or conflict resolved.
  7. Overbooking rules — behaviour when direct push overlaps OTA-sold dates.
  8. Rate strategy — can website keep independent direct rates while OTAs use CM rates?
  9. Supported OTAs — channel identifiers mapping to our Booking.source enum.
  10. ARI model — push vs pull for availability/rate updates.
  11. Stay date semantics — check-in/check-out exclusive vs inclusive (we use exclusive checkout for stays).
  12. Certified integration path — STAAH ChannelConnect vs native Axis Rooms API.
  13. SLA — payload limits, rate limits, webhook delivery guarantees.
  14. Integration contact + UAT timeline — test property setup and production cutover plan.
  15. Payment handling — how OTA-collected payments appear in payloads (we use Razorpay for direct only).

14. Ten-item checklist for AxisRooms meeting

Send this list directly to AxisRooms:

  1. Provide complete API documentation for reservation create, modify, cancel, and availability — plus sandbox credentials.
  2. Confirm authentication method (API key, Bearer token, IP restrictions) and rate limits.
  3. Deliver property mapping sheet for all Jade villas: Property ID, Room Type ID, Rate Plan ID — confirm one room type per villa with inventory = 1.
  4. Define inbound webhook contract (new / modified / cancelled reservation) including idempotency key field.
  5. Define outbound reservation API for pushing our direct website confirmed bookings to block OTA channels.
  6. Define cancellation / inventory-release API when we cancel a synced direct booking.
  7. Clarify overbooking behaviour when our push conflicts with an existing OTA reservation.
  8. Confirm rate strategy: independent direct website rates vs OTA rates managed only in AxisRooms.
  9. List supported OTA channels and the source identifier we should store per booking.
  10. Confirm whether integration uses STAAH ChannelConnect or a native Axis Rooms API, and assign an integration contact with UAT timeline.
