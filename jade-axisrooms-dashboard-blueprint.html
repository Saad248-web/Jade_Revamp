1. The two-way sync in one paragraph

Axis Rooms is a channel manager that sits between Jade and the OTAs (Airbnb, Booking.com). Jade never talks to OTAs directly. There are two directions that matter:

  • OUTBOUND (Jade → Axis): whenever a date becomes unavailable in Jade (OTA API 9 save, website pay, staff booking/block), Jade calls API 2 (one date range) then API 1 (daywise nights) so OTAs stop selling those nights. Price / restrictions use APIs 6, 7, 15.
  • INBOUND (Axis → Jade): when a guest books on an OTA, Axis POSTs to /api/webhooks/axisrooms (API 9); Jade validates, saves, then pushes inventory back.

Both directions are live in sandbox for hotel 1303. Remaining: CM confirmation of multi-night ranges, channel_managed flip per villa, production cutover (Sr. Waseem lead).

So yes: manual reserve, public booking, OTA API 9, and blocks push inventory to Axis for mapped villas. Multi-night stays use one API 2 request covering check-in → last occupied night (Vanessa, 16 Jul).

5. Editing a registered property → push changes to OTA

When staff edit a mapped villa (Quick Edit or Full Editor) and save, syncVillaChannelState() fires automatically:

What you editAxis API usedEffect on OTA
Price / ratesAPI 7 (bulk) → falls back to API 690-day price window pushed to OTAs
Set villa Live + bookableAPI 2 + API 15 (open)Inventory opened, stop-sale lifted
Set villa Hidden / Not bookableAPI 15 (close)Stop-sale on OTAs
Important limit (by Axis design): editing descriptive content — villa name, photos, amenities, description — does not push to OTAs. Axis has no API to edit an OTA listing's content. Only availability, price, and open/close sync. Listing text/photos on Airbnb/Booking.com are always maintained in those OTA portals by the Jade team. Jade content edits still update jaderetreats.com instantly.

6. Pulling OTA data into the dashboard (better clarity)

These read APIs would make the dashboard far more transparent. None are built yet — they're my recommendations:

Proposed API 12 — Pull Booking (safety net)

  • Nightly cron pulls OTA bookings in 10-day windows
  • Funnels through the same parser + idempotency as the webhook
  • Catches any push Axis retried and we missed → zero lost OTA bookings

Proposed API 13 — Connected OTAs

  • Lists which OTAs are connected + their integer ids
  • Unlocks APIs 3/4 (per-OTA block) which need those ids
  • Shows "Airbnb ✓ · Booking.com ✓" per property in the dashboard

No API API 5 — OTA Availability

  • Not activated in Axis sandbox for this channel
  • Removed from API 9 inbound path (caused UAT delays)
  • Do not require API 5 for booking ack

Dashboard clarity add-ons

  • Per-villa channel status badge (Website-only / Mapped / Live)
  • Axis sync log panel (audit log already records every push)
  • Booking Records already tags channel (Airbnb OTA, Booking.com OTA, Direct, Staff)

7. Housekeeping page — keep it (it does NOT need Axis)

Verdict: KEEP. Axis has no housekeeping/stay-status API — that's correct. But the Housekeeping page never needed one. It reads Jade's own bookings and shows arrival / in-house / departed / turnover / ready. Because OTA bookings land in Jade via API 9, the Housekeeping board already covers every channel (direct + Airbnb + Booking.com) automatically.
QuestionAnswer

Suggested enhancement: add a channel tag on each housekeeping card (Direct / Airbnb / Booking.com) so on-site staff know the guest's origin. Purely a Jade-side display change; no Axis dependency.

8. Custom new property behaviour — keep website-instant, OTA manual

A new villa created in the dashboard goes live on jaderetreats.com immediately. It will not auto-appear on OTAs — Axis has no create-property API. This is normal for every channel manager.

  1. 1 Create the villa in Jade → live on the public site right away.
  2. 2 Jade team creates the listing on each OTA portal (Airbnb, Booking.com).
  3. 3 Jade sends property/room/rate to Axis (CSV onboarding) → gets hotelId / roomId / ratePlanId.
  4. 4 Paste those IDs into the villa's Axis mapping → sync (inventory/price/restrictions) turns on.
Recommended UX (Option A + B from prior discussion): give each villa a channel mode — Website-only (default for new villas, skips Axis cleanly) vs Channel-managed (requires IDs, sync active) — plus a status badge and a short onboarding checklist shown when a custom property is added. Turns the manual OTA step into a guided flow.

9. Edge cases & double-booking prevention

The classic race: direct booking + Airbnb booking at the same moment

Edge caseHow Jade prevents / handles itMechanism
Two direct bookings, same villa & night, at once Only one succeeds; the other gets 409 Night Locks — unique index on {villaId, date}; second insert fails atomically
Direct + OTA booking overlap (the race above) OTA booking saved as CONFLICT, surfaced to staff — never lost Overlap check inside withTransaction() + night-lock check on inbound
Axis retries the same booking push Processed once; retries return success and do nothing WebhookEvent unique {eventId, source} (idempotency)
OTA cancels a booking Booking → cancelled, nights released, OTA can resell API 9 cancel branch → releaseNightLocks()
Inventory push to Axis fails (network/500) Auto-retried until success (max 10 attempts) axisrooms-retry cron (every 15 min)
Guest starts booking but never pays Pending booking expires, nights freed expire-pending-bookings cron + night-lock release
Loopback (OTA booking pushed back to the OTA) Prevented — inbound bookings are pre-marked synced axisRoomsSynced = true on API 9 records
Forged webhook call Rejected with 401 Timing-safe accessKey compare in the body
The honest limitation: no channel manager can make a same-second direct+OTA collision impossible — the OTA may accept a booking milliseconds before our close reaches it. The correct industry pattern (which Jade implements) is: detect the overlap, mark it CONFLICT, and let staff resolve rather than double-host a guest. To shrink the window: keep pushes fast (done) and add API 12 nightly pull as a backstop (proposed).

10. CRONs & WebhookEvent — plain-English

MechanismWhat it isWhy it matters
WebhookEvent A tiny record written the first time we see an OTA event ({eventId, source}, unique) Axis retries pushes until it gets "success". This makes retries safe — the same booking is never created twice.
Night Locks One row per villa per booked date, with a unique constraint The database itself guarantees a date can be held by only one booking — the core anti-double-booking guard.
axisrooms-retry cron Runs every 15 min; re-pushes any inventory sync that failed If Axis was briefly down, the close/open still lands. Max 10 attempts, then flagged for staff.
expire-pending-bookings cron Runs every 5 min; expires unpaid pending bookings Frees dates a guest reserved but abandoned, so inventory isn't stuck.
Done axisrooms-pull cron (daily) Nightly reconcile via API 12 Backstop that guarantees no OTA booking is ever missed, even if a webhook failed.

Cron scheduling note: Vercel Hobby allows only once-daily crons; the 5-/15-min jobs need Vercel Pro or an external scheduler hitting the URLs with CRON_SECRET.

  • Keep custom-property → website-instant behaviour; add the channel-mode flag + checklist for OTA onboarding.
  • Keep the full API reference doc (including "what Axis can't do") — it's documentation, nothing to delete.
